## 1. 学習塾営業支援WEBアプリケーションと学習塾基幹システムの生徒情報を同期するアプリケーション開発
### 概要
API経由で学習塾営業支援WEBアプリケーションから同期すべき生徒情報を取得。同期処理は学習塾基幹システムへのスクレイピングで行う。

### 期間
2022年5月 - 現在

### 開発手法
GitHub Issueによるタスク管理。ローカル環境で検証と実装を繰り返す。

### チーム
日本人エンジニア2名(上長と私)

### 役割
開発メンバー(私がメインで開発し、上長にサポートいただいていた。)

### 使用技術
Windows11 + WSL2 / Git / GitHub / JavaScript / Node.js / Playwright / Serverless / AWS Lambda

### 担当業務
- 営業チームと議論しながら、開発チーム用のシステム構成図とシーケンス図の作成
- 使用技術選定
- GitHub Issue作成
- Issueによる検証と実装の工程管理
- 実装

### 取り組み
#### 1. 手を早く動かし、実現性検証や検討事項明確化を早期に行えた。
- 選定した技術によるスクレイピング処理が可能かどうかをまず検証する必要があった。
- 検証作業を早く繰り返すことで、実現性確認と作業工程見積もりも早期に行うことができた。
- また検討事項を早期に明確化することができ、営業チームと議論を重ね、スムーズな実装が行えた。

## 2. 学習塾の運営業務を効率化する WEB アプリケーション開発
### 期間
(前半)2020年7月 - 2021年3月 / (後半)2021年7月 - 2022年4月

### 開発手法
GitHub Issueによるタスク管理。1 - 2週間を1スプリントとし、1スプリントにつき1回のリリース。

### チーム
- (前半) 日本人エンジニア2名(上長と私) / 外国人エンジニア6 - 8名
- (後半) 日本人エンジニア2名(上長と私) / 外国人エンジニア2 - 3名

### 役割
- (前半) 開発メンバー
- (後半) 開発メンバーリーダー

### 使用技術
Windows / Windows + WSL2 / Docker / Git / GitHub / Bitbucket

PHP / Laravel / MySQL / Bootstrap / CoreUI / JavaScript / jQuery / Node.js / Cypress

AWS (ECR / ECS / RDS / CloudWatch / EC2) / Sentry

### 担当業務
#### 新規機能開発とバグ改修
- 営業チームと共に要件定義やUI設計
- 開発チーム用の仕様書作成や更新
- Local / Staging / Production環境でのバグ調査
- GitHub Issue作成
- 新規機能実装 (単体テスト / 結合テスト / EndToEnd テスト実装含む)
- バグ改修 (単体テスト / 結合テスト / EndToEnd テスト実装や修正含む)
- 手動テスト用テスト項目作成や修正

#### 運用・保守
- Staging / Production環境へのデプロイ作業
- GitHub Dependabotから報告された脆弱性のあるパッケージを確認し、自動作成されたPRレビュー

#### チームマネジメント (開発メンバーリーダーとして)
- チームメンバーへタスクのアサインとタスク状況管理
- チームメンバーからのプルリクエストをレビュー (1st レビュー:私 / 最終レビュー : 上長)
- 月1でチームメンバーとの1on1

### 取り組み
#### 散在する脆弱 / 膨大 / 複雑ロジックを積極的にリファクタリング。後続タスクの時間を削減。
- タスクに関連のあるロジックは、型定義 / メソッド抽出 / クラス最小化などを積極的に行いました。
- タスク範囲外のロジックについては、コメント (TODO / FIXME) を残すようにしました。
- リファクタリングの際は、テストコードの修正と更新を行い、バグ混入の可能性を極力排除しました。
- レビューや slack チャットを通し、チームメンバーに対しても積極的リファクタリングを促しました。
- 後続する新規機能開発やバグ修正の障壁を低くし、後続タスクの時間を軽減することができました。

#### 上長からのレビューなどで得た学びをチームへ共有。レビューの負担を軽減。
- 私が作成したプルリクエスト(PR)は上長がレビューしていました。
- 他メンバーが作成したPRの1stレビューは私が、最終レビューを上長が担当していました。
- レビューやslackチャットを通して、日々の業務から得た学びをチームメンバーへ共有しました。
- その結果、チームの実装スキルが向上し、レビューにかかる時間が短縮しました。
- 特に上長による最終レビューでは、一発LGTMが頻繁に出るようになりました。

## 自己PR
### 1. ただコードを書くだけじゃない。
- こうしたらもっとユーザーにとって使いやすくなるのではないかという
- 大学と前職(デザイナー) で培った UI / UX視点での提案を行えることが強みです。
- 現職で携わったプロジェクトでは、タスク範囲内外関わらず積極的に提案をし、UI / UXの改善に貢献しました。
- 提案は
    - 1つの提案に対し、複数案とそれぞれのメリットデメリットなどをまとめ、
    - その後チームと議論を重ね、提案が通ればGitHub Issueを作成する流れで行っていました。

### 2. 働く環境や技術に対するキャッチアップの姿勢
- 外国人エンジニアとコミュニケーションを取る時は英語を使います。
- リモートワークでのコミュニケーションコストを削減するため、業務時間外で英語を勉強しました。
- また、業務 / Udemy / 本などから得た学びを、til(today i learned)としてまとめて、スキルアップをしました。
- そのtilを外国人エンジニアへ積極的に共有し、チームのスキルアップも図りました。

### 3. フルリモートでも業務を問題なく行えます。
- 未経験 WEB エンジニアとして入社してから現在に至るまで、フルリモートで働いていました。
- slackでのチャットと通話を使って、コミュニケーションを取っていました。
- slackチャットでは、
    - 質問や共有をする際は、背景と目的を必要に応じて記載。
    - 提案をする際には、複数の提案とそれぞれにおけるメリットデメリット、おすすめの解決策を記載。
    - 参考リンクや画像は、分かりやすくなる場合に必ず記載。
- slack 通話では、
    - 通話で議論すべき内容を事前にまとめ、通話前にチャットで送信。
    - 議事録はチャット上でリアルタイムでまとめ、不参加書も含めて確認できるように。
- 以上のことを意識し、リモートワークでも問題なくスムーズに業務を行うことができました。
